先将每个模块都用go mod init 创建一个go.mod文件（go modules机制要求每个模块的根目录都必须有一个go.mod文件，这样程序才能识别这是一个模块
每个目录下都只能有一个package包名！包名可以自定义，但是最好与目录名称相同。最终只有一个package main，func main()是最终的可执行文件

使用go modules方式管理模块的问题：ModuleB在文件中被引用了两次，如果需要更改路径就需要修改很多。

当我运行go run Demo.go时工作区自动解析依赖流程：
1、读取go.work文件，确定本地模块的位置
2、将导入路径（如github.com/YYYSSSRRR/HelloGO/ModuleB）自动映射到本地的./ModuleB
3、忽略go.mod中的replace指令

⚠️go.work的映射时如何实现的？

(1)定位go.work文件
(2)解析use指令中的每个本地路径，检查路径是否存在且是一个有效模块，获取声明的模块路径（如github.com/YYYSSSRRR/HelloGO/ModuleB），在内存中建立一个模块路径到本地文件的一个映射表
(3)在依赖解析时优先使用go.work本地映射，go.mod中的require和replace可以全部删除
go.work中的本地路径可以是未发布的代码，如果发布之后就可以直接import模块路径，正式开发环境中并不会受到go.work影响

将代码正式上传到远程仓库中

单仓库包含多个子模块：使用Git Tag+子目录路径
1、提交代码并打tag

Tag 必须是 子模块名/版本号 的格式！ 否则 Go 无法解析子模块。
在ModuleA中使用时，不需要replace，直接运行

为什么需要使用go mod tidy?
自动清除没用的依赖和添加漏掉的依赖（import但没写到go.mod中）
保证和go.sum完整一致
在引入新模块，删除import引用，升级依赖等时候要用

在其他项目中使用go get引用上传到github的模块

1、使用go mod init初始化创建go.mod文件
2、用go get拉取远程仓库的模块

自动在go.mod和go.sum中添加此模块
3、在文件中import，代码中就直接可以使用了
